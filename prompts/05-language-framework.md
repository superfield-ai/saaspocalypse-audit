# Category 5: Language, Framework & Tool Choices

**What this assesses:** Whether the technology choices introduce risk the team
understands and manages — or risk that is invisible because it was chosen by
an AI agent, copied from a tutorial, or simply never examined.

---

## What to look for in the scan

- What runtime, language, and framework are in use? Are they current,
  actively maintained, and receiving security patches?
- Are dependencies pinned to specific versions, or floating (e.g., `^1.2.3`,
  `>=1.0`, `latest`)?
- Are there known vulnerabilities in the dependency tree? (Look for a
  lockfile that can be scanned, or an existing audit output.)
- How many direct and transitive dependencies does the project have? A large
  dependency tree is a large attack surface.
- Are any packages obscure, recently published, or with low download counts?
  (Hallucinated package names from AI-generated code become real packages when
  attackers publish them.)
- Are the framework's built-in security features in use — CSRF protection,
  output escaping, parameterized queries — or are they bypassed?
- Is there a license audit? Dependencies with incompatible licenses create
  legal exposure.

## Signals of problematic tool choices

- EOL runtime (Node 14, Python 3.7, Ruby 2.x) — no longer receiving security
  patches
- Dependency manifest with no lockfile — installs whatever is latest
- Framework security middleware disabled or missing from configuration
- Raw string interpolation into SQL queries alongside an ORM that would
  handle this safely
- Packages with names that are slight misspellings of well-known libraries
- No evidence anyone has audited what the dependencies actually do

## Grading

**A** — Runtime and framework are current and will receive security patches for
at least two more years. All dependencies are pinned. Vulnerability scanning
runs in CI and blocks deployment on high-severity findings. The team has audited
the direct dependency list and has a policy for adding new dependencies. The
framework's security features are enabled and not bypassed. License compliance
is verified.

**B** — Runtime and framework are current. Dependencies are pinned. Vulnerability
scanning exists but does not block deployment. The team is generally aware of
what the major dependencies do. Some framework security features are bypassed
for convenience in specific places with acknowledged tradeoff.

**C** — Runtime or framework is approaching end-of-life or is already past it.
Dependencies are partially pinned. No automated vulnerability scanning. The
team has not examined the transitive dependency tree and does not know the full
attack surface. Framework security features are inconsistently applied.

**D** — Runtime or framework is end-of-life and no longer receiving security
patches. No lockfile — the application installs whatever is current at deploy
time. No vulnerability scanning. The dependency graph was generated by an AI
agent and has not been reviewed by a human who knows what each package does.

## Real-world incident anchor (use for C or D grades)

**Log4Shell, December 2021.**

Log4j is a Java logging library — a piece of infrastructure code used by
millions of applications to write log entries. It was present in applications
built by companies ranging from Apple to Amazon to the US government, usually
as a transitive dependency: a library that was included by another library,
which was included by another library, which the development team actually
chose.

A researcher discovered that Log4j would execute arbitrary remote code if a
specially formatted string was logged — and any application that processed user
input and logged it was potentially vulnerable. The attack string was eleven
characters: `${jndi:ldap://...}`.

The US Cybersecurity and Infrastructure Security Agency called it "the most
serious vulnerability in a decade." Hundreds of millions of devices were
potentially affected. Many organizations discovered they were running Log4j
only when they were breached.

The lesson is not that Java is dangerous. The lesson is that every dependency
you include — and every dependency of that dependency — is code that runs in
your application with your application's permissions. Not knowing your
dependency tree is not the same as not having one.

**AI-specific variant — ghost packages.**

AI coding tools sometimes generate import statements for packages that do not
exist. When a developer installs dependencies without reviewing the list, those
package names sit empty. Attackers monitor for such names and publish malicious
packages under them. The developer's next install silently downloads and
executes attacker-controlled code. This attack has been observed in the wild
and is growing in frequency as AI code generation becomes more common.

## Questions to ask if scan is ambiguous

- "Has anyone checked the list of libraries your system uses to make sure they
  are all still actively maintained and receiving security updates?"
- "Were any of the packages in your system chosen by an AI, and has a person
  verified that those packages actually exist and do what was intended?"
